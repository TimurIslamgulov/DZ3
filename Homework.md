# Первое использование контроля версий 
## Подготовка:
* Установливаем Git 
* Устанавливаем VS Code
### Первые шаги: 
1. Запускаем VS Code
2. Мышкой нажимаем "вид" -> "терминал" 
* теперь работаем с консолью
3. Проверяем версию ПО:
* git --version
* Если версия указана, продолжаем работу
4. Теперь нужно представиться программе, введя две команды:
* git config --global user.name "<ваше_имя>
* git config --global user.email "<адрес_почты@email.com>"

### Поздравляю, мы готовы к работе!

## Начало работы с проектом:
1. Создаём папку для работы над проектом
5. Делаем из папки репозиторий командой:
* **git init** 

 *Данная команда используется один раз для каждой папки, на всё время её отслеживания.* 
 
 *Репозиторий – папка, в которой настроена система контроля версий (хранилище файлов, поддерживающих версионность)*
# Список команд для работы (ver. 1):

* **git --version**

Проверяет версию программы
* **git init** 

Инициализация папки, более подробное описание выше*

* **git add < file_name >** 

Добавляет файл, в отслеживаемые

* **git commit -m <commit_name>**

Фиксация изменений,присвоение имени версии проекта

*У программистов это называется «закоммитить»*

*  **git commit –am<commit_name>**

Команда, которая объединяет: **git add + git commit -m<...>** - подходит только для уже отслеживаемых файлов. При использовании комитятся все файлы в папке, в кот. есть несохраненные в ранних версиях изменения.

* **git log** 

Выводит список последних коммитов

* **git reflolg** 

Выводит расширенный список коммитов

* **git log –p** 

Выводит последние коммиты и содержание изменений

* **git status** 

Статус изменений, если таковые имеются

* **git checkout <commit_numb>**  

Используется для передвижений между коммитами (не считая последнего)

* **git checkout master** 

Вернет на самый последний коммит

* **git diff** 

Позволяет смотреть отличия между версиями

* **git diff <commit_numb1> <commit_numb2>** 

*Показывает различия конкретных коммитов*


* **git rm <file_name>**

Используется для удаления файла из отслеживаемых ***и для физического удаления файла***

### Допустим нужно закоммитить много файлов:
1)	git add . (git add - a) 

 Добавляет все файлы, в отслеживаемые

2)	git commit –m“com_name”

Теперь все файлы закоммитили (зафиксировали текущее состояние)

*Далее Можно использовать **git commit –am<...>***

### Волшебная буква - q
Позволяет продолжить работу с терминалом, даже, если тот отказывается принимать команды. Ввёл "q", нажал "Enter" и всё как рукой сняло. Терминал продолжает принимать команды.

# Многопользовательскоt использование системы контроля версий:

Для добавления файла в игнорируемые создаём файл с расширением  **.gitignore** и вписываем туда файлы и папки, которые хотим игнорировать. фалы указываются с 
расширением, а папки с символом «/» на конце.

* **git branch** 

 Выводит список всех веток на экран
 
* **git branch < branch name >**
 
 Создаёт новую ветку

* **gut checkout < branch name >** 

Используется для перехода на другую ветку

* **git merge < branch name >**

Команда для слияния веток (команда вызывается из ветки, в которую нужно внести изменения из «черновика» (ветки, имя которой мы вводим))

* **git branch -d < branch name >**

Удаляет ветку

* **git branch -D < branch name >** 

Тоже удаляет ветку 

* **git log --graph**

Позволяет увидеть лог коммитов в виде дерева

### С помощью веток мы «параллелим» работу. (Имеется ввиду выполняем работу параллельно)

При разрешении конфликта есть вероятность потерять информацию. Поэтому слиянием веток и разрешением конфликтов занимается специальный человек (чаще всего старший разработчик или team lead).

Если конфликта нет, коммит появляется автоматически.

Если есть конфликт, сначала надо его решить, а потом коммитить.

* **git checkout –b < branch name >**

Команда для создания новой ветки с переходом на неё (соответственно со всем содержимым предыдущей)

* **git merge --abort**

Отмена слияния веток (используется до коммита)

# Работа с удалёнными репозиториями (Github)

## GitHub это сервис компании Майкрософт.

* **git clone <link>**

Позволяет скопировать внешний репозиторий на свой ПК.

## Для работы с удаленным репозиторием : 

1. Создать аккаунт на GitHub.com

2. Создать локальный репозиторий

3. “Подружить” ваш локальный и удалённый репозитории. 

 GitHub при создании нового репозитория подскажет, как это можно сделать

4. **git push** - отправляит (push) локальный репозиторий в удалённый (на GitHub), при этом, возможно, вам нужно будет авторизоваться на удалённом репозитории

5. Провести изменения “с другого компьютера”

6. **git pull** - выкачать (pull) актуальное состояние из удалённого репозитория

## Для первой отправки проекта на github используем команды:

1. **git remote add origin <ссылка>**

2. **git branch –m main** (переименовываем ветку мастер в ветку main)

3. **git push –u origin main** (отправляем на удалённый путь origin ветку main)

### Далее используем команды **git push и git pull**

# Последовательность действий для pull request
1.	**Fork**-*(Копируем репозиторий в свой аккаунт)*
2.	**git clone < link >** - Копируем репозиторий на свой ПК
3.	**git branch < branch name >** - *Создаём новую ветку, работаем только в этой ветке.*
4.	**Вносим изменения изменения + git commit -am"commit name"**
5.	**git push** - *Отправляем изменения на Github*
6.	*Нажимаем **pull request**
